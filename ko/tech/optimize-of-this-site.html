블로그 최적화 팁

✍🏼 작성일 2020년 10월 31일   
❗️ 참고: 이 글이 작성된 지 이미 일이 지났습니다. 시의성에 유의하세요

서문

이전에 이 블로그의 도메인 최적화 설정에 관한 몇 가지 설명]이라는 글을 쓴 적이 있습니다. 그때 사용한 몇 가지 방법은 더 이상 사용하지 않게 되었고, 동시에 다른 몇 가지 방법을 추가하여 이 Jekyll 블로그에 적용했습니다.
이 과정에서 주의해야 할 몇 가지 세부 사항이 있어 이 글을 작성하게 되었습니다.

CDN 서비스

이전에 사용하던 CDN 서비스는 Baidu CDN 서비스와 Qiniu의 정적 리소스 호스팅]을 이용하는 것이었습니다.
img.xheldon.com을 Qiniu Cloud로 리졸브한 후, Qiniu Cloud에 관련 정적 리소스를 추가했습니다. 하지만 잘 알려진 이유로 Qiniu Cloud의 이 무료 개인 CDN 서비스는 ICP 비안이 필요했고, 국내 물리 서버도 필요했기 때문에 이 방법을 포기했습니다.

오랜 시간 동안 저는 이미지와 정적 리소스를 직접 프로젝트에 포함시켰습니다. GitHub의 무료 저장소는 총 1GB의 용량 제한이 있기 때문에 가능한 한 텍스트로 설명하고 이미지는 최소화했습니다. 이는 상당히 제약이 많았습니다.
얼마 전 우연히 GitHub Pages를 기반으로 한 다른 블로그를 방문했는데, 그곳에서는 많은 이미지를 사용하고 있었고, 이미지 링크가 jsdelivr.net와 관련이 있는 것을 발견했습니다. 그래서 해외의 무료 CDN 서비스인지 추측했고, 조사해 보니 실제로 그렇다는 것을 알게 되었습니다. 게다가 국내에도 서버가 있어 무료로 가속화할 수 있었습니다.
GitHub 저장소의 리소스 앞에 고정된 경로를 추가하기만 하면 됩니다. 첫 요청 시 리소스가 S3에 저장되며, 이후 GitHub가 다운되더라도 정상적으로 접근할 수 있습니다. S3에 리소스가 없으면 GitHub의 내용을 직접 요청하는, 즉 CDN의 기본적인 동작 방식입니다.

저는 무료 CDN 가속 서비스가 가능하고, 복잡하지 않으며, jsdelivr의 주요 장점 중 하나가 중국에서 합법적인 ICP 라이선스를 보유한 해외 공공 CDN 서비스 제공업체라는 점에 매우 만족했습니다. 중국 내 수백 개의 서버를 보유하고 있었습니다.

바로 실행에 옮겼고, 다음과 같은 단계를 거쳤습니다:

전용 정적 리소스 저장소 추가

이 단계는 jsdelivr CDN 서비스가 정확하게 리졸브할 수 있도록 하기 위함입니다. 예를 들어, 로컬의 어떤 이미지 참조 경로가 /static/img/in-post/logo.png이라면, 그 고유 경로는 /img/logo.png입니다. 이 내용을 GitHub 저장소에 푸시한 후,
접근 경로는 https://github.com/Xheldon/x_blog-static/tree/master/img/logo.png가 됩니다. 제 사용자명은 xheldon이고, 저장소는 x_blog-static이므로, jsdelivr CDN 서비스를 이용한 접근 주소는
https://cdn.jsdelivr.net/gh/xheldon/x_blog-static/img/logo.png이 됩니다. jsdelivr 서비스에 등록할 필요도 없고, 아무것도 설정하지 않아도 사용할 수 있어 매우 훌륭합니다!

블로그 저장소 수정

주로 다음과 같은 단계가 있습니다:

  1. static 폴더를 삭제하고, 이 폴더를 새로 생성한 GitHub 리소스 저장소 x_blog-static에 푸시한 후 삭제합니다.
  2. 리소스 저장소 x_blog-static을 블로그 저장소의 서브모듈로 추가합니다: git submodule add https://github.com/Xheldon/x_blog-static.git ./static, 그리고 정적 리소스 저장소(방금의 static 폴더 내용)를 static 폴더 아래에 둡니다.
  3. _config.yml 파일을 수정하여 구성 객체 static_url: https://cdn.jsdelivr.net/gh/xheldon/x_blog-static를 추가합니다. 이는 다음 단계에서 정적 리소스의 절대 경로를 수정하기 쉽게 합니다.
  4. 로컬 개발 구성 파일 _config.dev.yml을 추가하고, 구성 객체 static_url: /static를 추가합니다. 이는 다음 단계에서 정적 리소스의 절대 경로를 수정하기 쉽게 합니다.
  5. 로컬 절대 경로로 참조된 정적 리소스를 전역적으로 수정합니다. 예를 들어 img의 src가 /static/img/in-post/logo.png 또는 /static/css/main.css 등의 경로라면, `https://static.xheldon.cn/img/logo.png` 등으로 교체합니다. 여기서 주의할 점은, Liquid 문법 태그 `{{}}`는 markdown 파일에서도 사용할 수 있다는 점입니다. 매우 유용합니다.
  6. 자주 수정되는 파일들, 예를 들어 main.scss 파일은 여전히 로컬에 두는 것이 좋습니다. 이유는 아래에서 설명합니다.
  7. 수정 사항을 원격 저장소에 푸시합니다.

알려진 문제

몇 가지 문제를 해결한 후, 또 다른 문제가 발생했습니다. Jekyll은 scss를 지원하기 때문에, 로컬에서 resoruce/css/main.scss에 sass 문법을 작성한 후, html에서 resoruce/css/main.css(파일 확장자 주의)를 직접 참조할 수 있습니다. Jekyll은 자동으로 scss를 css로 변환합니다. 하지만 이렇게 하면 해당 스타일 파일은 반드시 로컬에 저장되어야 하며, 원격 저장소에 둘 수 없습니다.

이러한 점을 고려하여, 자주 수정되는 파일들(예: css, js 등)은 블로그 저장소에 직접 두고, 정적 리소스 저장소에는 더 이상 포함시키지 않기로 했습니다.

또한, 로컬 개발 시 기본 default layout 템플릿에서 Liquid의 전역 변수(예: static_url)를 정의하고, 글의 markdown 파일에서 `{{static_url}}/img/logo.png`와 같이 사용하려고 했지만, 실험해 보니 불가능했습니다. 조사해 보니 Liquid, Jekyll, GitHub Pages 모두 site 또는 pages, 혹은 paginator 수준의 변수 정의를 지원하지 않는다는 것을 알게 되었습니다.

그래서 default 레이아웃 파일에서 assign 변수를 직접 정의하고, default 레이아웃을 사용하는 post 레이아웃의 마크다운 파일에서 접근해 보았지만 여전히 실패했습니다. Jekyll이 지원하는 환경 변수 JEKYLL_ENV=dev를 사용해 보기도 했지만, 앞서와 마찬가지로 레이아웃 간(어떤 것이 중첩되든) 변수를 공유할 수 없어 포기했습니다. 조사해 본 결과, Jekyll은 먼저 글을 렌더링한 후 레이아웃 파일을 렌더링하기 때문에 마크다운 파일이 외부 레이아웃의 변수를 가져올 수 없는 것이 원인이었습니다.

사실 각 마크다운 파일 앞에 템플릿 코드를 추가해 jekyll.environment 값을 확인한 후 static_url 같은 변수를 할당하고, 마크다운에서 해당 변수를 사용하는 방법도 가능했습니다. 하지만 모든 마크다운 파일의 헤더에 이런 난해한 로직을 추가하는 것은 원치 않아 포기했습니다. 마찬가지로 static_url 변수를 사용하는 모든 곳에서 environment를 확인하는 방법도 있었지만, 예를 들어 이미지를 삽입할 때마다 `{{if environment=dev 'static' else 'https://jsdelivr.net/xxx/'}}/img/logo.png` 같은 식으로 작성해야 하는 것은 더 번거로워 포기했습니다.

따라서 최종 해결책은 두 개의 Jekyll yml 설정 파일을 작성하는 것이었습니다. 하나는 기본 _config.yml로 Github Pages에서 온라인 페이지 생성 시 사용하고, 다른 하나는 _config.dev.yml로 로컬 개발 시 사용합니다. 둘의 유일한 차이점은 static_url라는 설정 값으로, _config.dev.yml에서는 /static 값을, _config.yml에서는 https://cdn.jsdelivr.net/gh/xheldon/x_blog-static 값을 갖습니다.

한 가지 주의할 점은 배포 후 일부 이미지에서 403 오류(Package size exceeded the configured limit of 50 MB. Try https://github.com/xxx instead)가 발생할 수 있다는 것입니다. 따라서 최종적으로 버전 태그나 브랜치 이름을 추가해야 합니다. 저는 브랜치 이름(https://cdn.jsdelivr.net/gh/xheldon/x_blog-static@master)을 사용했습니다.

Gitalk 추가

Gitalk은 Github Pages 전용 댓글 도구로, css와 js를 추가하고 token을 설정하면 사용할 수 있습니다. 새 글이 작성될 때마다 작성자가 페이지를 한 번 열면 해당 GitHub Pages 저장소의 issue에 현재 글 제목으로 된 issue가 자동 생성됩니다. 이후 글 아래에 댓글이 달리거나 issue에 댓글이 달리면 글로 표시되는 매우 편리한 도구입니다. 여기서도 jsdelivr의 cdn 서비스를 사용했으며, 이는 공식적으로 지원됩니다.

페이지 레이아웃 수정

글을 볼 때 왼쪽에 큰 공백으로 프로필 사진과 네비게이션이 계속 표시되는 것이 공간 낭비라고 느껴졌습니다. 하지만 홈페이지, 프로젝트 페이지, 프로젝트 분류 페이지에서는 프로필 사진, 네비게이션, 검색 기능이 필요했습니다. 따라서 브라우저 너비가 1080px 이상일 때는 네비게이션을 왼쪽에 표시하고 검색창을 보여주기로 했습니다. 다른 경우에는 네비게이션을 상단에 표시합니다.

또한 글 페이지에서는 화면 너비가 1080px 이상일 때 table of content(목차)를 표시해 사용자가 원하는 섹션으로 빠르게 이동할 수 있도록 했습니다. 이는 외부 사이트 사용자가 페이지의 특정 앵커로 바로 링크할 수 있게 하고 SEO에도 도움이 됩니다.

글 단락 제목 수정

SEO를 고려해 한 페이지에는 h1 태그 하나만, h2 이상의 태그는 여러 개 사용해 검색 엔진이 쉽게 인식할 수 있도록 했습니다.

프로젝트 페이지 추가

개인 프로젝트 전용 페이지를 추가해 제가 참여한 모든 프로젝트를 찾을 수 있도록 했습니다. 일부는 외부 링크로 연결되고, 일부는 개인 설명 프로젝트입니다.

A Simple Comparison Between Slate and ProseMirror

✍🏼 Written on Jan 6, 2023    💡 Updated on Jan 9, 2023
❗️ Note: it has been days since this article was written, please be aware of its timeliness
🖥  Note：A Brief Comparison Summary After Two Years of Using Both Slate and ProseMirror
📚  Also published on Craft： https://www.craft.do/s/WCeEgxNhxwF2cu

Having used PM for less than two years, I’ve developed large modules like smart tables; and having used Slate to this day, I’ve gone through major version upgrades from 0.47 to 0.5+, so I feel somewhat qualified to speak on the topic. Since many in the community often ask me about the differences between the two, I decided to write a blog post to briefly summarize them.

Slate Pros:

  1. Quick to learn, works out of the box, with easy-to-grasp concepts and readable code, especially for those familiar with React. Its view layer is entirely React-based, making it possible to get up to speed within a week.

  2. Highly customizable. You can modify core modules like slate and slate-react to extend plugins and architecture according to your business needs. Modifying the underlying code is relatively easier compared to PM.

  3. Well-suited for collaborative scenarios like Sharedb. Its JSON-based design seems almost tailor-made for Slate.

Slate Cons:

  1. The author is slow to fix bugs, as he doesn’t work on it full-time. Maintenance largely depends on the community. Some APIs are clearly flawed (e.g., .include, which has tripped me up), while others throw exceptions or return null inconsistently. The author often asks for PRs instead of fixing issues himself.

  2. Unstable APIs. It’s still in the 0.x version and hasn’t reached a stable release, so future architectural changes could be significant.

  3. The official version is practically unusable, especially for CJK language users dealing with input method events. You’ll need to download and modify it yourself.

  4. Performance optimization is challenging. Since the view layer is entirely React-based, optimizing it largely depends on optimizing React, which isn’t inherently fast.

  5. Poorly written code can easily crash the editor, leaving it unusable. There’s no robust error-handling mechanism beyond modifying the framework code.

  6. Unclear data flow can lead to bugs. For example, you can directly assign editor.children, overwriting existing editor content without any warnings.

  7. Cursor and Mark issues are hard to hack. The code feels beginner-level (in my opinion).

  8. Poor documentation, with some parts inconsistent with actual API behavior.

  9. Tightly coupled with React.

  10. Mobile support isn’t great and requires modifications.

ProseMirror Pros:

  1. The author maintains it full-time and is very responsive to community questions.

  2. It’s been in stable release for over three years, with consistently stable APIs.

  3. Backed by big companies—Google uses the author’s other project, CodeMirror (e.g., in Chrome DevTools’ Elements tab)—so it’s highly credible.

  4. Clear data flow (though hard to understand at first).

  5. Less prone to crashes than Slate. Errors won’t render the entire editor unusable, making it more stable.

  6. The author is a freelancer who accepts paid bug fixes, charging €200 per hour (I emailed him; that’s the rate if I recall correctly).

  7. The document tree model is very useful. Its strict Schema constraints and text/token-based positioning system are brilliant, though they make collaboration tricky.

  8. Not tied to any view framework. You can use React, Vue, or Angular for the view layer, though custom components are rarely needed. For basic elements like paragraphs, headings, and lists, you only need to describe the DOM structure via the API.

  9. Good mobile support.

ProseMirror Cons:

  1. Some concepts and code are hard to grasp (the author’s technical depth feels unfathomable).

  2. Modular and not out-of-the-box, requiring some setup effort to get running.

  3. Implementing collaborative solutions is harder compared to Slate. The author opened an issue in 2017 promising a collaboration demo but still hasn’t delivered.

  4. Steeper learning curve, often taking two weeks or even a month to start developing basic features.

  5. Since it implements its own view layer (though developers can use APIs to render custom components like React), debugging framework-level view issues is tricky.

  6. Modifying the source code is nearly impossible due to high costs, so fixes usually depend on the author.

Summary

Slate is suitable for large companies because it’s quick to get started (large companies rely on input-output ratios—otherwise projects can’t secure resources. How do you explain to leadership if after two months it’s still just a demo?). Thus, results can be delivered rapidly, but refining it to perfection requires significant manpower for deep customization, making it ideal for large companies: quick adoption early on, followed by adding personnel for heavy modifications. Additionally, its extensibility is exceptional—after customization, it can meet almost any requirement (for example, we’ve implemented various modules and features like inline comments, hierarchical heading folding, embedding third-party modules, etc.).

Prosemirror is better suited for small and medium-sized companies. It has decent cross-platform support, and if bugs arise, you can confidently rely on the author to fix them—or even pay them to resolve issues if absolutely necessary. Although the learning curve is steeper, a core developer at a small or medium-sized company should be able to produce a demo after about two weeks of exploration.
